# 0. 一段话看懂这项工作

PX4 允许经典控制栈和学习型低层控制器在同一飞控软件中共存，并在飞行中交接执行器控制权。问题定位在**控制权移交事件及其与控制器能力的交互**：相关要求散落在文档、消息协议和参考实现中；飞行模式切换成功无法保证旧控制器已经退出、新控制器已经独占接管；物理失控又可能同时受到目标控制器、移交机制、控制历史和测试 harness 的影响。

我们将这些隐式要求重构为一份四条款契约——**准入、移交、残留、回退**——并提出 BATON：用可执行检查器检测契约违约，用模式转换图和状态化 fuzzing 搜索危险交接条件，用保持—切换差分发现额外物理风险，再根据命中的条款选择机制、目标控制器、触发方式和历史差分完成归因。

当前已经确认两个 PX4 上游问题：准入回复结构存在未初始化字段；一个时间戳回归使外部模式的 allocation 配置持续被判为陈旧，经典 `control_allocator` 在学习模式激活后仍继续发布，与学习控制器形成无告警双写。行为实验表明，在经典控制器能够恢复的候选状态下，部署态 `mc_nn_control` 切入分支可出现灾难性失控。2026-07-12 完成的 harness 硬化将 wall-clock timer 改为 sim-time/lockstep 后，关键锚点达到 **100/100 一致复现**，状态收敛速度提升约 **85×**；此前约 0.78 的一致率主要由测试时间基准污染解释。该结果使修复前后机制差分具备了可执行的确定性基础。

# 1. 研究对象与基本测试单元

## 1.1 主要测试对象

本文的主要测试对象是：

> **生产飞控中异构控制器之间的执行器控制权移交软件子系统。**

它跨越：

- Commander / Nav State；
- Flight Mode 与 External Mode Registration；
- mode requirements、guard 和配置回复；
- 经典控制模块与学习型控制器的启停；
- `control_allocator` 与 `actuator_motors` 的写者所有权；
- PID 积分器、滤波器和神经网络隐状态的生命周期；
- 主动回退与 failsafe 强制转换；
- ULog、协议 trace、仿真时间与闭环物理结果。

经典控制器、`mc_nn_control` 和 RAPTOR 是移交参与者，也是闭环行为对象。经典控制器可充当反事实参照，其内部状态同样进入审计范围。

## 1.2 基本测试单元

一次控制权移交事件表示为：

```text
T = (source, target, trigger, handover_state, history, environment)
```

- `source`：当前控制器或控制流水线；
- `target`：拟接管的控制器或控制流水线；
- `trigger`：飞手命令、地面站、任务逻辑或 failsafe；
- `handover_state`：候选交接时刻的姿态、角速度、速度等；
- `history`：此前经历的控制器、模式和切换序列；
- `environment`：风、模型偏差等条件。

有些飞行模式变化只改变任务或 setpoint 来源，并不改变低层执行器写者。本文只把**真正改变或试图改变执行器控制权**的事件纳入核心测试空间。

## 1.3 状态表示

飞行模式表达控制平面的意图，执行器写者集合表达数据平面的事实。测试状态至少记录：

```text
State = (flight_mode, control_pipeline, actuator_writers, controller_memory)
```

理想状态：

```text
flight_mode = Neural Control
actuator_writers = {mc_nn_control}
```

已观察到的异常状态：

```text
flight_mode = Neural Control
actuator_writers = {mc_nn_control, control_allocator}
```

因此，**模式切换成功并不代表控制权已经完成移交。**

# 2. 为什么这是一项软件工程问题

## 2.1 隐式、跨层的软件契约

控制权移交没有集中式规约。相关要求散落在：

1. 官方文档；
2. 消息和配置协议；
3. Commander / ModeManagement 实现；
4. 学习控制器参考实现；
5. 相关控制器论文中的安全承诺。

同一条移交链会穿过控制平面、消息层、模块启停、执行器数据面和物理闭环。局部代码可以看似合理，组合行为仍可能违约。

## 2.2 书面锚点与现实对撞

契约来自可追溯的书面承诺，不是研究者凭空设定的标准。

### A2：allocation 应被关闭

集成文档说明，学习型/外部低层控制模式可通过配置关闭 control allocation，使目标控制器直接输出执行器指令。

现实链路：

```text
config 仅发布一次
→ 新鲜度判断回归
→ config 持续被判 stale
→ safe default 保持 allocation 开启
→ control_allocator 与 learned controller 双写
```

### A4：RAPTOR 声称与经典控制完全解耦

RAPTOR 集成材料使用“completely decoupled”等表述。当前日志却显示学习模式期间仍存在经典来源电机帧。该对撞直接支持移交条款的可证伪性。

### “Use at your own risk”的边界

实验性免责声明覆盖学习控制器本身的性能风险。外部模式注册、ModeManagement、guard 槽位、配置传递和执行器写者管理属于 PX4 核心替换设施。本文审计的是这套核心设施能否兑现其明确的接口和配置语义。

## 2.3 现有测试手段的缺口

- 只看飞行结果：能发现翻机，难以知道哪条软件保证被破坏；
- 只看模式状态：无法知道真实执行器写者；
- 只读控制器代码：看不到远端配置回归和 allocator 竞争；
- 只跑一次场景：测试时间基准可能制造虚假随机性；
- 只比较经典与神经：只能定位切换复合操作的额外风险，无法完成机制因果归因。

# 3. 控制权移交契约

## 3.1 四类条款

| 条款 | 核心问题 | 主要对象 |
|---|---|---|
| **准入 Admission** | 当前状态、数据和目标控制器是否满足接管条件？ | guard、mode requirements、数据新鲜度、控制器准备状态 |
| **移交 Handover** | 旧写者是否退出，新写者是否完整、及时、独占接管？ | config、模块启停、allocator、uORB 写者、时序 |
| **残留 Residual State** | 前一阶段的内部状态是否污染下一阶段？ | PID 积分器、滤波器、GRU hidden state、setpoint 历史 |
| **回退 Fallback** | 切回经典或安全模式后，控制权和物理状态是否恢复？ | Commander、failsafe、经典栈、恢复时间、高度损失 |

## 3.2 当前证据状态

| 条款 | 当前发现 | 状态 |
|---|---|---|
| 准入 | `mc_nn_control.cpp:207` 的 `can_arm_and_run` 恒真；RAPTOR 主要检查观测新鲜度；guard 词汇缺少姿态/角速度表达；回复结构未完整初始化 | ✅ 代码级问题；未初始化路径为上游缺陷 |
| 移交 | config freshness 回归使 allocation 配置持续 stale；经典 allocator 不退出；`actuator_motors` 双写；约 11% 电机帧来自经典栈；全链无告警 | ✅ 已确认机制缺陷；物理因果贡献待修复差分 |
| 残留 | 经典 `_vel_int` / `_rate_int` 跨模式存活；RAPTOR GRU 在激活边沿重置 | ◐ 生命周期差异已确认，物理后果待测 |
| 回退 | 主动 `N→C` 与 failsafe 强制边缺少系统实验 | ⚠️ 测试空白 |

## 3.3 问题的结构性特征

1. **无声**：stale 配置回退到默认值，双写和未初始化字段均不会自动形成清晰错误；
2. **跨层**：根因位于时间戳比较或消息结构，症状出现在电机输出和姿态失控；
3. **责备错置**：实验性学习控制器会吸收对物理异常的第一反应；
4. **时序敏感**：错误时间基准可改变注入时机、状态收敛和重放结果。

# 4. 模式转换图与场景范围

模式转换图用于组织测试空间：

- 节点：`(flight_mode, control_pipeline, actuator_writers)`；
- 边：真正改变或试图改变执行器控制权的事件；

- 连续负载：姿态、角速度、速度、风、延迟；
- 路径历史：此前经历的控制器和切换序列。

覆盖单位：

```text
edge × handover-state cell × trigger × history class
```

## S1：主动切入

```text
Classical → Learned
```

当前大规模实验主要集中在该边。现有约 926 次对齐评估落在 commanded 切入及其相邻设置中。

## S2：主动回退

```text
Learned → Classical
```

用于测试经典安全网能否在不同失稳程度下重新接管。

## S3：failsafe 强制转换

```text
Learned → Return / Land / Classical safety pipeline
```

用于测试无人选择交接时机时的准入、配置和恢复行为。

## S4：连续切换

```text
Classical → Learned → Classical → Learned ...
```

用于测试积分器、滤波器和 GRU 状态的残留与累积效应。

S1 的现有证据构成完整图中的第一块；S2–S4 是由契约直接推出的待测区域。

# 5. 方法总览：发现—定位—归因

BATON 的流程为：

```text
契约重构与可追溯形式化
          ↓
模式图制导的状态化场景生成
          ↓
基础保持—切换差分发现额外风险
          ↓
契约检查器定位违约条款
          ↓
条款制导的针对性差分诊断
          ↓
代码根因 + 物理因果贡献
```

三个组件承担不同职责：

- **Fuzzing**：寻找危险边、状态、触发和历史组合；
- **检查器**：指出哪一条契约被破坏；
- **诊断差分**：判断具体因素是否造成或放大物理后果。

# 6. 契约重构与可执行检查器

## 6.1 人工重构流程

从固定 commit 的文档、消息、代码和参考实现中提取：

```text
来源 → 原始承诺 → 形式化属性 → 可观测信号 → 违约条件
```

每个检查器必须携带来源与版本，形成“来源—条款—信号—规则”的可追溯表。

## 6.2 检查器类型

“契约检查器”包含：

- ULog 后处理监视器；
- 协议 trace 监视器；
- 插桩后的运行时检查器；
- 离线状态扫描器；
- 静态初始化卫生检查；
- 主动行为一致性测试。

部分条款无法仅依赖普通 ULog，需要 trace、插桩或静态分析。

## 6.3 候选检查器

| 编号 | 条款 | 规则 | 证据性质 |
|---|---|---|---|
| M1 | 移交 | 声明关闭 allocation 后，allocator 必须静默 | **无后见可定义**：直接来自 A2 |
| M2 | 移交 | 激活时使用的 config 满足实现中的新鲜度语义 | **诊断后精化**：精确形式带后见性 |
| M3 | 移交 | armed 正常控制期间 `actuator_motors` 满足单写者 | **无后见可定义**：来自独占控制权语义 |
| M4 | 准入 | 接收方消费的回复字段必须显式初始化 | **无后见可定义**：消息卫生原则 |
| M5 | 准入 | guard 对相关状态变化产生非平凡响应 | 需明确控制器适用域规约 |
| M6 | 回退 | ConfigOverrides 声明与实际 failsafe 行为一致 | 需主动实验 |
| M7 | 残留 | 重新激活前后内部状态满足规定生命周期策略 | 需插桩和策略定义 |

## 6.4 回扫主张的边界

历史回扫可以说明检查器能在已有日志中发现违约。关于“传统手段漏掉它”或“第一条日志即可发现”的强主张，只由 M1、M3、M4 等无后见组支撑。M2 的精确形式用于根因诊断，不承担无后见检测能力的证明。

# 7. 基础行为差分：发现额外风险

## 7.1 统一原则

对于任一转换边：

```text
共同前缀 + 同一候选交接状态
参照臂：保持 source
实验臂：source → target
```

参照臂无需制造 `source → source` 的伪切换。

## 7.2 经典到学习控制器

```text
参照臂：C stays C
实验臂：C → N
```

该差分说明：

- 候选状态对经典控制器仍可恢复；
- 执行“切换并由 N 接管”这一复合操作引入额外风险。

该结果无法单独区分目标控制器恢复包络、移交机制、二者交互和 harness 时序。

## 7.3 学习控制器到经典控制器

```text
参照臂：N stays N
实验臂：N → C
```

解释矩阵：

| N 保持 | N→C | 含义 |
|---|---|---|
| 失控 | 恢复 | 回退有效 |
| 失控 | 失控 | 回退过晚、移交失败或超出经典恢复包络 |
| 稳定 | 失控 | 回退过程引入风险 |
| 稳定 | 稳定 | 回退可行 |

# 8. 核心方法：契约条款制导的差分诊断

## 8.1 基本原则

1. 基础行为差分发现异常；
2. 检查器定位疑似违约条款；
3. 根据条款选择最小诊断差分；
4. 通过修复、重置或触发对照判断因果贡献；
5. 避免没有明确诊断问题的全组合实验。

## 8.2 六类差分

### D-A：行为差分

```text
C stays  vs  C → N
N stays  vs  N → C
```

回答：切换复合操作是否引入额外风险，或回退是否改善结果。

### D-B：目标控制器差分

```text
C → mc_nn  vs  C → RAPTOR
```

回答：共享移交设施和相近交接状态下，不同目标控制器是否表现不同。

该差分支持 oracle 区分力和控制器选择性后果。RAPTOR 的“同一条件”结论仍需完成独立写者归因 E4，确认其实际数据面配置与 `mc_nn` 对齐。

### D-C：机制差分

```text
原始 PX4：C → N
修复移交机制：C → N
```

可修复或控制：

- config freshness；
- allocator silence；
- 单写者；
- 回复初始化；
- handover sequencing。

回答：已确认的软件机制缺陷是否造成或放大物理失效。

β 结果已经提供 100/100 的确定性锚点，该差分现已具备执行条件，应列为近期最高优先级实验。

### D-D：触发方式差分

```text
commanded source→target
vs
failsafe-triggered source→target
```

回答：failsafe 是否采用不同 guard、配置或时序路径。

### D-E：历史/残留差分

```text
干净单次切换 vs 连续多次切换
保留内部状态 vs 重置/重新对齐
```

回答：控制历史和内部状态残留是否改变后续结局。

### D-F：harness / 时间差分

```text
wall-clock timer vs sim-time / lockstep timer
```

回答：测试系统时间语义是否污染注入时机、状态收敛、重放和性能标签。

该差分已经完成并落在 β：历史 0.78 一致率被消除，锚点达到 100/100 一致复现。

## 8.3 条款到差分的选择规则

| 命中条款/现象 | 首选诊断差分 | 目标结论 |
|---|---|---|
| 准入允许高风险状态 | 立即切换 vs 延迟/拒绝至满足 guard | 状态相关准入能否避免风险 |
| allocator 未静默 / 双写 | 原始 vs 修复单写者 | 移交缺陷的物理因果贡献 |
| 仅一个目标控制器失控 | `C→mc_nn` vs `C→RAPTOR` | 控制器选择性后果 |
| 连续切换才异常 | 单次 vs 连续；保留 vs 重置 | 残留状态因果作用 |
| failsafe 独有异常 | commanded vs failsafe | 触发路径差异 |
| 回退后仍失控 | `N stays` vs `N→C` + 写者恢复时间 | 回退有效性和时机 |
| 结果重复性差 | wall-clock vs sim-time + 重复执行 | harness 污染识别 |

# 9. 三层归因链

## 9.1 契约归因

回答：哪项保证被破坏？

例如：

- guard 恒真；
- allocation 配置未生效；
- 执行器双写；
- 回退后的经典写者没有及时恢复。

由可执行检查器完成。

## 9.2 代码根因归因

回答：哪条实现链路造成违约？

手段包括：

- 源码追踪；
- commit history / bisect；
- trace 和插桩；
- 静态初始化检查；
- identity gate。

双写链路已定位为：

```text
一次性 config
→ freshness 比较回归
→ config 持续 stale
→ safe default 保持 allocation 开启
→ allocator 与 learned controller 双写
```

## 9.3 物理因果归因

回答：该软件缺陷是否造成或放大闭环失控？

依赖机制差分：

```text
原始机制：物理结果 J0
修复机制：物理结果 J1
```

解释：

- 修复后异常消失：机制缺陷具有主要因果作用；
- 修复后异常减弱：机制与控制器/状态存在交互；
- 修复后异常保持：协议缺陷真实存在，当前物理失效仍需从控制器能力或其他因素解释；
- 修复后出现新异常：修复改变了时序或状态生命周期，继续诊断。

## 9.4 联合解释矩阵

| 契约违约 | 行为差分 | 修复后 | 可下结论 |
|---|---|---|---|
| 有 | 无 | — | 软件协议缺陷存在，当前未显示物理危害 |
| 无 | 有 | — | 控制器/状态风险，或检查器覆盖不足 |
| 有 | 有 | 异常消失 | 机制缺陷具有主要因果作用 |
| 有 | 有 | 异常减弱 | 机制与控制器/状态交互 |
| 有 | 有 | 异常保持 | 协议缺陷和行为失效共存，因果联系待继续分析 |
| 两控制器均违约 | 仅一个失控 | — | 共享平台缺陷产生控制器选择性后果 |
| 连续路径异常 | 重置后恢复 | — | 状态残留具有因果贡献 |

# 10. Harness 硬化与测试有效性

## 10.1 已完成的 β 判定

旧 harness 使用 ROS wall-clock timer 驱动关键 offboard 行为，PX4/SIH 使用仿真时间。两套时间基准错位会改变：

- 交接触发所对应的动力学状态；
- setpoint 和传感器更新的相对次序；
- 状态收敛速度；
- 双写消息的交错；
- 固定场景的重放一致性。

2026-07-12 完成 sim-time/lockstep 硬化后：

- 关键锚点达到 **100/100** 一致复现；
- 状态收敛速度提升约 **85×**；
- 先前约 **0.7817** 的成对一致率主要由 harness 时间污染解释；
- 当前没有证据支持该翻转为 PX4/控制器的内生随机性质；
- 并行执行和机制修复差分已解锁。

## 10.2 对既有结果的影响

### 仍然成立

- F1 的灾难差分锚点在硬化 harness 下得到更强复现；
- 准入和移交代码级问题仍然成立；
- 双写和未初始化字段属于独立软件缺陷；
- 经典保持与部署态 `C→mc_nn` 的行为差分仍然成立。

### 需要降级或重做

- 旧密扫中“高维、非凸、带洞、确定性边界”等形状主张全部退役；
- 旧数据只能表述为：**硬化前数据呈现非单调迹象**；
- 风、姿态、角速度和 delay 的净化边界需要在 sim-time harness 上重新刻画；
- 重复执行现在属于防御性测试实践，用于确认新区域和新配置的稳定性，不再承担“系统天然随机”的论证。

## 10.3 方法学启发

> CPS fuzzing 的时间语义属于测试构造的一部分。错误时间基准可以生成虚假非确定性、污染边界标签，并阻塞机制归因。

该发现构成独立的软件测试方法学贡献：测试工具必须显式审计仿真时间、外部节点时间、注入调度和收敛条件。

# 11. 当前行为证据的准确解释

## 11.1 F1：确定性灾难差分锚点

在同一候选状态：

```text
C stays C：经典控制器恢复
C → mc_nn：部署态神经分支灾难失控
```

硬化 harness 后关键锚点 100/100 复现。

准确结论：

> 候选状态仍处于经典控制器恢复能力内；执行“切换并由部署态 mc_nn 接管”引入了稳定可复现的额外物理风险。

当前仍待机制差分回答：单写者修复后，该风险保留、减弱或消失。

## 11.2 F2：搜索能力与旧边界数据

guided search 曾比 random/grid 更快定位可疑区域，但 3/3 强确认档存在平局，完整边界主要来自后续密扫。

当前处理：

- 保留“fuzzer 能较快定位候选危险区域”的有限主张；
- 预注册净化评估指标：time-to-first-confirmed-failure、危险状态格覆盖率、有效 eval 数、种子稳定性；
- 旧非单调形状仅作为探索线索；
- 若净化比较无法显示稳定优势，搜索效率 RQ 可从论文核心中移除。

## 11.3 B1/B2/B3：范围定位

三个负结果支持：

> 主要行为异常定位在切换事件及其瞬态附近，稳态风、估计污染和普通渐进行为退化无法解释现有灾难锚点。

这些结果不排除目标控制器特异性。目标控制器能力、移交机制和二者交互均保留为 RQ3 的候选解释。

## 11.4 RAPTOR：第二目标控制器

现有对齐 campaign 中 RAPTOR 未出现同等级失控，去除输入裁剪后仍保持稳定。

当前作用：

- oracle 区分力；
- 第二真实控制器上的 harness 可迁移性；
- 控制器选择性后果的初步证据。

“完全相同移交条件”的强表述等待 E4 写者归因闭合，确认 RAPTOR 的实际写者、配置和双写状态与 `mc_nn` 实验可比。

# 15. 软件工程贡献

## C1：问题抽象

把异构控制器的执行器控制权移交定义为独立的软件测试对象，以四条款契约和模式/所有权联合状态描述其语义。

## C2：可追溯检查器

将人工重构的承诺操作化为日志、trace、状态扫描和静态检查器，并区分无后见检测与后见诊断。

## C3：契约条款制导的差分诊断

用保持—切换差分发现风险，用命中条款选择机制、目标控制器、触发和历史差分，形成契约归因、代码根因和物理因果贡献三层链条。

## C4：真实 PX4 缺陷与系统行为

报告准入回复未初始化、config freshness 回归、allocator 不退出和执行器双写；报告硬化 harness 下 100/100 复现的部署态 `C→mc_nn` 灾难差分，并通过机制修复实验进一步归因。

## C5：CPS 测试 harness 有效性

证明 wall-clock 与 sim-time 混用可制造虚假非确定性、污染标签并阻碍因果诊断；给出 lockstep、收敛门和重放审计协议。

## C6：状态化场景生成（条件贡献）

若净化评估成立，报告转换图制导 fuzzing 在危险状态定位上的效率；若优势不足，该部分作为实现机制保留，不承担主要贡献。

# 18. 摘要草案（现实证据版）

> 学习型控制器正通过飞行模式和插件机制进入生产级自驾仪，使经典与学习型控制流水线能够在运行时交接执行器控制权。我们发现，这次交接所依赖的要求散落在文档、消息协议和参考实现中，缺少统一规约与自动检查；模式切换成功也无法保证执行器控制权已经完成独占移交。我们将这些要求重构为控制权移交契约的四类条款——准入、移交、残留和回退，并设计 BATON：以可追溯检查器检测无声违约，以模式转换图组织状态化场景 fuzzing，以保持—切换差分发现额外物理风险，再根据命中的条款选择机制、目标控制器、触发方式和历史差分完成三层归因。在 PX4 上，我们确认了准入回复中的未初始化字段，以及一个使 allocation 配置持续陈旧、经典 allocator 无法退出并与学习控制器双写的上游回归。在经典控制器能够恢复的候选状态下，部署态 `mc_nn_control` 切入分支出现灾难性失控；将测试 harness 从 wall-clock 修正为 sim-time/lockstep 后，关键锚点达到 100/100 一致复现，表明此前的结局翻转主要来自测试时间语义污染。当前工作正通过单写者修复前后差分、主动与 failsafe 回退、连续切换和状态重置实验，区分移交机制、目标控制器和控制历史对物理后果的因果贡献。
