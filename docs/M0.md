# M0: RAPTOR Closed-Loop Bring-Up and Oracle Sanity

## Scope

M0 ran a real SITL experiment only:

- classical takeoff to hover/hold,
- in-flight switch to RAPTOR external mode,
- ULOG capture and topic sanity,
- literal missing-setpoint oracle sanity.

No M1 work was done: no ROS 2 offboard task node, no MAVSDK failure injection, no metric pipeline, no controller divergence harness, and no search.

## Simulator and Board

Primary simulator: SIH, using `px4_sitl_raptor_sih` and `sihsim_quadx`.

The reproducible board overlay is tracked at:

```text
boards/px4/sitl/raptor_sih.px4board
```

Install it into the ignored PX4 tree with:

```bash
./scripts/install_raptor_sih_board.sh
```

Build evidence:

```bash
./scripts/build_px4_raptor_sih.sh
```

Output: `docs/m0_raptor_sih_build.log`.

Note: `make px4_sitl_raptor_sih sihsim_quadx` builds the target but its generated SIH run command starts PX4 from `build/.../src/modules/simulation/simulator_sih`, where `etc/init.d-posix/rcS` is not visible. The M0 run uses the working direct launch from `build/px4_sitl_raptor_sih`:

```bash
PX4_SIMULATOR=sihsim PX4_SIM_MODEL=sihsim_quadx ./bin/px4 .
```

This preserved SIH for the actual M0 flight instead of falling back to Gazebo.

## Reproduce

From the host shell:

```bash
sg docker -c 'cd /mnt/nvme/uav_sf && CONTAINER_NAME=uav_sf_m0_build ./docker/run.sh bash -lc "cd /workspace && ./scripts/build_px4_raptor_sih.sh"'

sg docker -c 'cd /mnt/nvme/uav_sf && CONTAINER_NAME=uav_sf_m0_run ./docker/run.sh bash -lc "cd /workspace && python3 -m pip install --break-system-packages pymavlink pyulog numpy -q && ./scripts/m0_run_experiment.sh"'
```

The run script sets:

```text
NAV_DLL_ACT 0
COM_DISARM_LAND -1
IMU_GYRO_RATEMAX 250
COM_RC_IN_MODE 4
COM_RCL_EXCEPT 8
MC_RAPTOR_ENABLE 1
MC_RAPTOR_OFFB 0
MC_RAPTOR_INTREF 0
```

It writes the M0 logger profile into `build/px4_sitl_raptor_sih/etc/logging/logger_topics.txt`, starts MicroXRCEAgent, starts SIH, runs `commander takeoff`, waits for hover, and switches RAPTOR via MAVLink `MAV_CMD_DO_SET_MODE`.

## Known Issues

- The generated `make px4_sitl_raptor_sih sihsim_quadx` run target has a startup working-directory issue. The direct `./bin/px4 .` launch from the build root works and was used for M0.
- The inherited RAPTOR SITL config logs a non-blocking `vision_target_estimator` and `landing_target_estimator` conflict, already present in Phase 1 smoke output.
- Logger reports "Too many subscriptions" for extra actuator/status instances, but still logs the requested eight M0 topics.
- `gcs_connection_lost` is true in headless SIH, but `NAV_DLL_ACT=0` prevented a data-link-loss action and `failsafe` stayed false.

## Oracle Sanity

The missing-setpoint oracle did not trigger active motor NaNs or disarm. `actuator_motors` total NaNs were only on unused quadrotor channels `control[4]` through `control[11]`; active channels `control[0]` through `control[3]` had zero NaNs.

Conclusion: current guarded RAPTOR holds on stale/missing setpoints. Finding NaN/instability cases needs later search-guided scenario generation, not a hand-picked missing-setpoint test.
