# M1 Oracle MVP

M1 的目标是把 oracle 主线跑通：同一个固定 theta 分别驱动 classical Offboard 和 RAPTOR external mode，抽 ULOG 指标，输出 classical-vs-RAPTOR 差分和四象限判定。本轮没有实现 NSGA-II、MAP-Elites、grid/random sampling 或任何 theta 空间自动搜索。

## Repository Artifacts

- `config/px4/init.d-posix/airframes/10046_sihsim_x500_v2`: tracked SIH-X500 v2 airframe.
- `scripts/install_m1_sih_x500.sh`: installs the airframe into the ignored PX4 tree and generated build tree.
- `config/m1_anchor_*.json`: manually selected fixed theta definitions.
- `scripts/m1_offboard_task.py`: ROS 2 parameterized task node.
- `scripts/m1_metrics.py`: ULOG metric extraction and per-run safe classification.
- `scripts/m1_compare.py`: differential four-quadrant comparison.
- `scripts/m1_diff_runner.py`: fixed-theta runner for classical + RAPTOR.
- `scripts/m1_inject_failure.py`: MAVLink `MAV_CMD_INJECT_FAILURE` wrapper.
- `docs/m1_diff_<tag>.json` and `docs/m1_diff_<tag>_summary.md`: machine/human-readable M1 results.
- `docs/m1_<tag>_<controller>.ulg`: ULOGs for the fixed theta runs.

## SIH-X500 v2

SIH itself is still the simulator, but the generic quad parameters are replaced by an X500 v2 approximation:

- mass and inertia from `external/PX4-Autopilot/Tools/simulation/gz/models/x500_base/model.sdf`: `SIH_MASS=2.0`, `SIH_IXX=0.0216667`, `SIH_IYY=0.0216667`, `SIH_IZZ=0.04`.
- Quad-X actuator geometry from `ROMFS/px4fmu_common/init.d-posix/airframes/4001_gz_x500`: `CA_ROTOR*_PX/PY/KM`.
- classical rate tuning from the X500 v2 airframe family: `MC_ROLLRATE_*` and `MC_PITCHRATE_*`.
- thrust model from gz x500 motor constants: `SIH_T_MAX=8.54858`, `SIH_Q_MAX=0.136777`, `MPC_THR_HOVER=0.60`.

The installer is called by `scripts/build_px4_raptor_sih.sh` and by `scripts/m1_diff_runner.py --skip-build`, so the ignored PX4 tree is reproducible from tracked files.

## RAPTOR Frequency

The SIH-X500 airframe and theta files set `IMU_GYRO_RATEMAX=400`. Startup logs for M1 show:

```text

INFO  [mc_raptor] Setting force_sync_native = 400 Hz / 100 Hz = 4
```

The previous M0 warning about `IMU_GYRO_RATEMAX=250` not being an integer multiple of the 100 Hz training frequency is gone in the M1 logs.

## Offboard Task

`scripts/m1_offboard_task.py` is a ROS 2 Jazzy `rclpy` node using `px4_msgs`. It publishes:

- `/fmu/in/offboard_control_mode`
- `/fmu/in/trajectory_setpoint`
- `/fmu/in/vehicle_command`

It subscribes to the versioned PX4 output topics exposed by the current DDS bridge:

- `/fmu/out/vehicle_status_v4`
- `/fmu/out/vehicle_local_position_v1`

The same theta setpoint stream drives both controllers. Classical runs switch to Offboard with `MAV_CMD_DO_SET_MODE` main mode `6`; RAPTOR runs switch to external mode 23 with main mode `4`, submode `11`. Mission event timing is based on elapsed PX4/DDS timestamps from PX4 output topics rather than wall-clock sleeps. The ULOG metric pipeline maps those task event offsets back onto ULOG boot-time using the first matching nav-state activation.

## Metrics

`scripts/m1_metrics.py` reads `.ulg` with `pyulog`, aligns the commanded `trajectory_setpoint` window, and emits one JSON result per run. The safe classification currently uses:

- tracking error max/RMS/final against commanded setpoint.
- roll/pitch maximum and angular-rate maximum.
- active motor `control[0..3]` NaN count.
- motor saturation ratio.
- disarm and failsafe flags in the mission window.
- local-position altitude floor.
- task completion / final error.

`scripts/m1_compare.py` computes RAPTOR-minus-classical deltas, time-to-divergence from interpolated position traces, and the four-quadrant label:

- `boring_both_safe`
- `interesting_not_bug`
- `primary_bug`
- `too_hard_not_bug`

Only `primary_bug` means classical safe and RAPTOR unsafe.

## Anchor Conclusion

The planned finite bad-setpoint anchor did not separate classical safe from RAPTOR unsafe. Inspection of `mc_raptor` explains why simple step amplitude is weaker than expected: `src/modules/mc_raptor/mc_raptor.hpp` sets `max_position_error=0.5` and `max_velocity_error=1.0`, and `mc_raptor.cpp` clips the observation before policy inference. The stale/missing-setpoint guard from M0 is still real, but the raw finite position jump is also bounded before it reaches the policy observation.

The manual backup `anchor_intref_lissajous` also did not produce a primary bug. Console logs confirm `MC_RAPTOR_INTREF=1`, `mc_raptor intref lissajous A=6 B=6 fa=4 fb=3 duration=5 ramp=0.5`, and `internal reference activated`, but the run remained safe by the implemented safety metrics. One caveat: internal RAPTOR reference is not republished as the public `trajectory_setpoint` topic, so this run is a vehicle-safety probe and mode-path check rather than a precise internal-reference tracking metric.

RAPTOR external mode 23 does follow the moving finite setpoint path under `MC_RAPTOR_OFFB=0`: the RAPTOR runs accept `nav_state=23`, the ULOGs contain the moving `trajectory_setpoint`, and the vehicle position/tracking metrics change with the commanded sine/step profiles. No fallback to `MC_RAPTOR_OFFB=1` was required.

## Failure Injection

`scripts/m1_inject_failure.py` wraps `MAV_CMD_INJECT_FAILURE` with PX4 unit/type names matching the `failure` command family. The theta schema has `faults`, `environment`, and `sensor_perturbations` fields, but the M1 anchor path does not depend on failure injection.

Current support matrix:

| class | SIH status in M1 | note |
|---|---|---|
| wind | supported by SIH params | use `SIH_WIND_N` / `SIH_WIND_E`, not the failure plugin |
| mass/inertia | supported by SIH params | `SIH_MASS` / `SIH_IXX` / `SIH_IYY` / `SIH_IZZ`; mass/inertia scaling was used in heavy theta runs |
| motor failure | wrapper implemented, simulator effect not validated | PX4 accepts the command path only when failure injection is enabled; official docs require simulator support and cite Gazebo Classic support |
| sensor/RC/MAVLink failures | wrapper implemented, SIH effect not validated | left as P1 follow-up; not used by M1 primary evidence |

Because the main M1 oracle path uses setpoint perturbations, this limitation does not block the fixed-theta differential runner.

## Known Issues

- ROS DDS output topic names are versioned in this environment, so the task node uses `vehicle_status_v4` and `vehicle_local_position_v1`.
- PX4/DDS timestamps include a timesync offset relative to ULOG boot-time. Metrics use relative task timing and align it to ULOG via the first matching controller nav-state.
- Logger emits `Too many subscriptions` warnings for extra topic instances when `logger_topics.txt` asks for all instances; instance `0` topics needed by M1 are present.
- A short `trajectory_setpoint turned stale` warning can appear before the ROS task fully owns the mission stream; final metrics use the mission window after controller activation.
- The inherited non-blocking `vision_target_estimator` / `landing_target_estimator` conflict remains visible, as in M0.
- The 10 m step attempt can trigger infrastructure-level offboard/failsafe behavior and is not a clean primary-bug anchor.
- RAPTOR internal-reference tracking is not currently aligned to a public commanded-setpoint ULOG topic; use external setpoint theta for strict tracking-error oracle evidence.

## Stop Point

M1 stops here. The metric and differential runner are ready for M2, but no search, optimization, sampling loop, repeated campaign, baseline, or failure-classification statistics were implemented in this commit.
