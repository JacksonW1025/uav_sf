# M2.6 Unshielded Observation Channel Probe

M2.6 targeted the RAPTOR observation fields that are not clipped in `mc_raptor.cpp`: `orientation` and `angular_velocity`. This pass did not run full M2b MAP-Elites and did not start M3.

## Mechanism

The selected mechanism is `IMU_GYRO_CUTOFF`, the PX4 low-pass filter on `vehicle_angular_velocity`. PX4 documents and code show this affects angular velocity sent to controllers, not EKF attitude. Therefore this M2.6 pass attacks the unshielded `angular_velocity` observation channel, not the orientation channel.

The pollution is fair for this target because both controllers consume the same shared topic:

- classical multicopter rate control subscribes `vehicle_angular_velocity`;
- RAPTOR copies that same topic into `raptor_input.angular_velocity` as `[x, -y, -z]`;
- each theta sets identical `boot_px4_params` and `px4_params` for both runs.

The scan combines this shared rate-channel lag with high thrust-to-weight ratio by setting `SIH_T_MAX`/`SIH_Q_MAX` and compensating `MPC_THR_HOVER`.

## Result

Confirmed primary bug count remains zero for this M2.6 pass. The tested unshielded angular-rate channel pollution, including 5 Hz gyro cutoff with TWR 2.3, did not produce a clean "classical safe, RAPTOR unsafe" opening.

M2.6 did produce secondary continuous-divergence evidence:

- high TWR alone (`30 Hz`, TWR `2.3`) made RAPTOR track worse while both stayed safe: classical tracking RMS `0.420 m`, RAPTOR tracking RMS `0.787 m`, quality `2.295`.
- high TWR plus moderate rate lag (`12 Hz`, TWR `2.3`) stayed both-safe but remained above the updated floor: classical tracking RMS `0.403 m`, RAPTOR tracking RMS `0.484 m`, quality `0.309`.

The stronger 8 Hz and 5 Hz rate-lag points did not monotonically increase RAPTOR-vs-classical divergence in this SIH setup.
