## 0. 为什么需要这份清单:模式转变（先读这段）

**前面所有 FUZZ 轮次是"手挑极端角落"模式。** FUZZ-1/1c 都是角落优先,3 个 eval 就命中,**MAP-Elites/NSGA-II 根本没用上**,搜索循环被绕过。

**"全量 scenario fuzzing"是另一种模式:搜索机器本身和它的有效性,变成了论文要 claim 的东西(RQ2 搜索有效性 + C3 消融)。** 这个转变决定了下面所有任务——很多在手挑模式下能"事后手查"或"绕过"的东西,在几千次运行的规模下必须先做成自动管线、且必须有可比基线。

**一句话排序(详见 §7):** Tier 0 是真阻塞(没 oracle 实现 + 标定就跑不了第一个 eval);Tier 1 的 baseline 和算力预算是最容易拖到太晚的两块;Tier 2 的蜕变 harness 该现在并行起。

## 1. Tier 0 — 硬阻塞（跑第一个 property-差分 eval 之前必须存在）

> **最容易被误判的一点**:`oracle_map_and_property_set_v0_2.md` 的 oracle 是**设计,不是实现**。§2.6 那张实现清单**一个框都没打勾**。在这四项做完之前,性质-差分搜索跑不了——搜索没有信号。

### 1.1 实现 `property_oracle.py`（判别器 + fitness 的本体）
- **现状**: 不存在。
- **要做**: 从 ULOG(真值回匹配,Method A)算 P1–P7 的鲁棒值 ρ_i(P8 后续)。接现有 `m1_metrics.py` 的 ULOG 读取约定。ρ 的复合算子语义见 oracle 文档 §2.2。
- **完成判据**: 给定一条 ulog,能输出 7 个 ρ_i 数值;在一条已知翻机的 ulog 上 ρ₁/ρ₂ ≤ 0,在标称 ulog 上全部 ρ_i > 0。

### 1.2 标定 16 个 `⟨CAL⟩` 阈值
- **现状**: 全是占位符。阈值没定 → ρ 算不出有意义的值 → 搜索被噪声带偏。
- **要做**: 按 oracle 文档 §2.3 跑标称(无扰动)轨迹、**两个控制器都测** ρ_i,阈值 = 噪声地板 × 安全因子(×3–5),一次填完 `θ_max, τ_rec, ω_max, u_sat, ε_sat, W_sat, Δu_max, ε_set, T_set, W_hold, s_min, A_max, W_osc, ε_ss, W_ss, margin_c`,落 `docs/oracle_calibration.md`。
- **完成判据**: 16 个值全部填实;标称 run 两控制器 ρ_i 都舒服 > 0(留出安全因子)。

### 1.3 接入去噪
- **现状**: 没接。风扰/传感器噪声会误导搜索方向。
- **要做**: PGFuzz 式——用控制算法里的**参考态** + **moving average** 滤噪后再算 ρ,统一进 `property_oracle.py`。
- **完成判据**: 标称 run 的 ρ_i 时间序列噪声显著低于阈值的安全因子余量。

### 1.4 差分用法包装接进 `m1_compare.py`
- **现状**: 没接。
- **要做**: 两控制器 ρ_i → S 级 → 差分判定条件(oracle 文档 §2.5:`ρ_i(neural) ≤ 0 ∧ ρ_i(classical) ≥ margin_c`)。注意:差分是**用法**,severity = 性质违反等级。
- **完成判据**: 喂入路线A 那 4 对已知 ulog,自动判出 classical=S0 ∧ mc_nn=S3 的差分(复现已知严差分)。

## 2. Tier 0.5 — θ 基因型具体化 + 搜索循环验证（紧随 Tier 0）

> FUZZ-1 是手挑角落,**从未真正定义过搜索空间**,也**从未在规模上跑过 NSGA-II/MAP-Elites**。全量铺开前这两件必须做。

### 2.1 把 θ 从"A/B/C 档描述"写死成可搜 genome
- **现状**: narrative 里 θ 是三个档位的文字描述;GA 要的是一个具体变量向量。
- **要做**: 定义每个搜索变量——是什么、连续/离散、上下界、哪些是 MAP-Elites 的分箱维度(扰动类型 × 幅度)。覆盖范围参考 θ 三档,但优先级见 §6。
- **完成判据**: 一份明确的 genome 规约;变异/交叉算子能在边界内产生合法场景。

### 2.2 把 fitness = `min_i ρ_i` 真正 wire 进搜索
- **现状**: "oracle 即 fitness"是 PGFuzz 的设计,你还没跑过它。
- **要做**: 搜索器从每次 eval 的 `property_oracle.py` 输出取 `min_i ρ_i`(或多目标按 P 类分目标)作 fitness,推场景向违反。
- **完成判据**: 搜索器能读到 fitness 并据此选择下一批场景。

### 2.3 搜索循环冒烟测试（放大前必做）
- **现状**: 搜索循环基本未验证。
- **要做**: 在一个**已知有信号的小 θ 子空间**上跑 NSGA-II/MAP-Elites,确认:fitness 在下降、多目标前沿在动、分箱在被填。确认无误再放大到全 θ。
- **完成判据**: 小规模跑出 fitness 单调改善 + 前沿/分箱有进展的曲线。

## 3. Tier 1 — 让结果可辩护的基建（容易拖到太晚,现在就 wire）

### 3.1 RQ2 baseline / 消融——**最常见翻车点**
- **风险**: 跑完花哨搜索拿到结果,才发现没可比基线;回头重跑时配置已 drift。
- **要做**: 在**同一 θ 空间、同一 oracle、同一评估预算**下,与主搜索**共用 harness** 地跑:random search、grid/uniform、(可选)corner-first 启发式。这是 C3"朴素搜索搜不到"消融的本体。
- **完成判据**: 基线与主搜索共用同一 eval 入口和 oracle;一条命令能切换搜索策略、其余全等。**从第一天起共用,不留到最后。**

### 3.2 算力预算 + 并行 + checkpoint
- **风险**: 每 eval = 2 次 SITL,CPU-only,每个 finding 要多种子复现。全量 = 几千 eval × 多种子,可能几百到上千 SITL 小时。**没有并行方案可能跑不完。**
- **要做**:
  - 估算: eval 数 × 单 eval 时长 × 总 wall-clock。
  - 并行: SIH headless/lockstep 确定性 → 理论可多容器并行。**验证同时跑 N 个容器是否稳定/无串扰**(你验证过吗?)。
  - checkpoint/resume: fuzz1c 有续跑,全 campaign 也要(断点续跑、已评估场景不重跑)。
- **完成判据**: 一个 wall-clock 估算 + 一个验证过的 N-容器并行方案 + campaign 级 resume。

### 3.3 日志规模纪律
- **风险**: 工作区已有 ~161 个未跟踪运行日志、远端历史 68 个旧 ulog;全 campaign 会再生成一个数量级 ulog(全 gitignored)。三千次运行后找不回某个 finding 的证据。
- **要做**: 定好——ulog 存哪、留多久、按什么命名/索引能从"θ + 种子"回查到具体 ulog。
- **完成判据**: 一套命名/索引规则;能用 (θ, seed) 反查到对应 ulog 路径。

### 3.4 把有效性纪律（§4.5）从"手查"变成"管线自动执行"
- **风险**: 手挑模式下能逐轮手查的清单,三千次运行里手查不可能。两条尤其要进管线:
  - **(i) 去基建污染自动化**: fuzz1c 那次是事后手工重分析 8 条 ulog。全量必须把"对称去污染"做进自动判定——offboard failsafe / nav RTL 落地 / 切换低于最小可恢复高度,**自动剔除、对两个控制器对称施用**(判据见 narrative §4.2 卫生机制 3 + §5.3 去污染重判)。
  - **(ii) 控制器 ID 自动 assert**: 每个 mode-23 run 自动确认在飞 mc_nn(228 Hz 推理、`network_output`=`actuator_motors`、`raptor_input` 不存在),否则 confound 悄悄混进结果。
  - 其余沿用: 真失败非 harness(`run_error`/console fault scan/到 `mission_end`/触发须真发火/fail-loud)、多种子复现、reachability 诚实。
- **完成判据**: 去污染 + ID 确认进自动判定;每个 finding 自带"非 harness + 多种子 + NN 已确认 + 已去污染"的自动标记。

## 4. Tier 2 — 并行科学轨道（不阻塞性质搜索,但该现在起）

### 4.1 蜕变 / 对称 harness（类型 2,**贡献承重项**）
- **为什么现在起**: oracle v0.2 里这是最 novel 的一块、且是解释非单调带的工具。不阻塞性质-差分 campaign,但**应和 oracle 实现并行起步**,否则论文里它会沦为脚注(见 oracle 文档 §1.7 实现顺序 vs 贡献权重)。
- **最小可跑版**: 任务脚本加"绕偏航旋转 / 镜像 setpoint + 风向"一档 + 响应等变残差比对;先在标称场景验证经典等变残差地板,再瞄准 FUZZ-1c 非单调带。
- **完成判据**: 能对一个场景跑"原始 vs 变换"两次、算等变残差;标称下经典残差接近地板(确认 oracle 有效)。

### 4.2 受控密扫非单调带（RQ3）
- **为什么**: narrative 已诚实标注 FUZZ-1c 那条带 wind/rate/approach 连变,分不清流形孔洞 / 高维边界 / 风敏感。
- **要做**: 单独跑**固定其它、单变切换姿态、每点多种子**的密扫,把那条带的结构定下来。和蜕变 oracle 互证(若是对称破缺,把结构观察升级成机制)。
- **完成判据**: 一张干净的一维姿态扫(其余固定)+ 每点多种子,能说出该带是哪种结构。

## 5. 条件阻塞（取决于全量搜索覆盖哪些 θ）

- **若 A 档走 shim 注入假状态信念(velocity/角速率)**: `patches/px4/m2b_state_shim.patch` 的 **patch drift 必须先整理**(external 树与 tracked patch 已 drift,反向 apply 失败)。纯 offboard/groundtruth/ulog-重分析的轮次**不受影响**;仅 shim 路线会卡。
- **若覆盖电机/传感器故障 θ**: SIH 对故障注入支持**未验证**,这类须路由到 Gazebo;且 **Gazebo plant 不对称**(RAPTOR 线踩过,-50% 两者都崩 = too_hard,非差分)。开工前定好哪些 θ 在哪个仿真器,并验证 SIH 的故障支持边界。

## 6. 方向性决策:新 oracle 改变了"该搜哪里"（写 prompt 时要传达给搜索）

崩溃 oracle 和性质 oracle 指向**不同的 θ 高产区**,别沿用手挑模式的直觉:

- **把预算分给 A 档(状态估计污染 / 物理失配)+ 稳态飞行。** 在崩溃 oracle 下它们看似是空的,但性质 oracle 下它们**最可能点亮新 bug**:预测的稳态漂移(P7)、极限环(P6)、抖振(P4)都活在持续侧风、挂载偏离训练值、估计污染这些条件里,**不在切换瞬态那个角落**。
- **别只盯 B 档切换带。** 路线A 的严差分出自 B 档,但那是灾难类(P1/P2),稀有;宽差分的主产区在 A 档 + 稳态。
- **C 档(setpoint 幅度)已被 GATE-3 判死**(mc_nn 对幅度比经典更鲁棒),不用再投预算。

## 7. 依赖与排序总览

| 阶段 | 内容 | 阻塞性 | 何时做 |
|---|---|---|---|
| **Tier 0** | oracle 实现(1.1)+ 标定(1.2)+ 去噪(1.3)+ 差分包装(1.4) | **硬阻塞**:不做跑不了第一个 eval | 最先,顺序内可并行 1.1↔1.2 |
| **Tier 0.5** | θ genome(2.1)+ fitness wire(2.2)+ 搜索冒烟测试(2.3) | **硬阻塞**:不做无法规模化搜索 | 紧随 Tier 0;2.3 须在放大前 |
| **Tier 1** | baseline/消融(3.1)+ 算力并行(3.2)+ 日志纪律(3.3)+ 有效性自动化(3.4) | **软阻塞**:不做结果不可辩护/跑不完 | **与 Tier 0/0.5 并行 wire,别拖到最后**;3.1 尤其要第一天共用 harness |
| **Tier 2** | 蜕变 harness(4.1)+ 密扫(4.2) | 不阻塞性质搜索 | **现在并行起**(4.1 是贡献承重项) |
| **条件** | shim drift / 故障仿真器路由 | 仅当覆盖对应 θ | 视 θ 覆盖而定 |

**关键路径**: Tier 0 → Tier 0.5 → 全量搜索;Tier 1(尤其 3.1 baseline + 3.2 预算)必须与关键路径**同时** wire,否则要么结果不可辩护、要么跑不完。Tier 2 独立轨道,越早起对论文越好。

**全量搜索可以正式开跑的总判据**: Tier 0 四项 + Tier 0.5 三项完成判据全绿,且 Tier 1 的 baseline(3.1)与并行/resume(3.2)就位。
