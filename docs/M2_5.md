# M2.5 Blocker Unlock Pass

M2.5 targeted the two blockers found after M2: missing shared state-estimator pollution dimensions, and `PX4_SIM_SPEED_FACTOR>1` runs that ended before the ROS offboard task reached `mission_end`. It did not run a full MAP-Elites campaign and did not start M3.

- `scripts/m1_diff_runner.py`: supports per-theta `boot_px4_params`, logs estimator/groundtruth topics, validates PX4 param command names, and gives PX4 enough sim-time before scripted shutdown at speed factors above 1.
- `scripts/m1_offboard_task.py`: records `PX4_SIM_SPEED_FACTOR`, effective wall timer rate, and supports `setpoint.max_wall_timer_hz`.
- `scripts/m2_map_elites.py`: adds `A_estimator` genome dimensions for shared EKF/GNSS delay, GPS noise/gates, EKF output predictor time constants, and IMU lever arm.

- `config/m2_5_speed_nominal_step.json` and `config/m2_5_speed_hover.json`: speed-factor consistency smoke thetas.

## A1 Shared Estimator Pollution

The active PX4 `3042f906` path for the old `EKF2_GPS_DELAY` knob is `SENS_GPS0_DELAY` / `SENS_GPS1_DELAY`; the PX4 source migrates the legacy import name to those sensor delay parameters. M2.5 therefore uses:

| family | params | status |
|---|---|---|
| GNSS delay | `SENS_GPS0_DELAY`, `SENS_GPS1_DELAY`, `EKF2_DELAY_MAX` | SIH-OK |
| GNSS weighting/gates | `EKF2_GPS_P_NOISE`, `EKF2_GPS_V_NOISE`, `EKF2_GPS_P_GATE`, `EKF2_GPS_V_GATE` | SIH-OK |
| output predictor lag | `EKF2_TAU_VEL`, `EKF2_TAU_POS` | SIH-OK |
| IMU lever arm | `EKF2_IMU_POS_X/Y/Z` | SIH-OK |

These parameters affect PX4 EKF outputs consumed by both controllers, not a RAPTOR-only observation path. `boot_px4_params` are appended to the per-run SIH airframe before PX4 starts, and the same values are also present in `px4_params` for console traceability.

Fairness evidence was generated for every scan point. Example:

- `docs/m2_5_estimator_harsh_1x_20260624/m2_5_fairness_m2_5_estimator_harsh_300ms_sine8hz.json`
- verdict: `fair_shared_estimator_pollution=true`
- effective params matched in classical and RAPTOR ULOGs, including `SENS_GPS0_DELAY=300`, `SENS_GPS1_DELAY=300`, `EKF2_DELAY_MAX=300`, `EKF2_TAU_VEL=1.0`, `EKF2_TAU_POS=1.0`
- RAPTOR `raptor_input` was active and finite; the same run had local-position vs groundtruth velocity error RMS `0.260 m/s` in RAPTOR and `0.560 m/s` in classical.

## B1 Speed-Factor Work

The original 4x failure mode is fixed: PX4 no longer shuts down before the task reaches `mission_end`.

Evidence:

- `docs/m2_5_estimator_smoke4x_20260624/`: 120 ms delay, `PX4_SIM_SPEED_FACTOR=4`, completed classical and RAPTOR, `mission_end=true`, elapsed wall `106.4 s`.
- `docs/m2_5_estimator_smoke1x_20260624/`: same theta at 1x, elapsed wall `165.2 s`.
- `docs/m2_5_speed_hover_4x_20260624/`: minimal hover at 4x, both controllers reached `mission_end`.

Strict speed-invariance did not pass the existing M2 noise floor. Even the minimal hover comparison had a few deltas above floor:

| theta | controller | examples above floor |
|---|---|---|
| `m2_5_speed_hover` | classical | tracking max `0.125 m` vs floor `0.05`; roll/pitch max `1.20 deg` vs floor `1.0`; angular-rate max `0.0216 rad/s` vs floor `0.02` |
| `m2_5_speed_hover` | RAPTOR | tracking max `0.154 m` vs floor `0.05`; final error `0.151 m` vs floor `0.05`; angular-rate max `0.0282 rad/s` vs floor `0.02` |

Conclusion: 4x is now usable for smoke/triage throughput, but primary-bug confirmation should remain at 1x until speed-factor result invariance is root-caused or the noise floor is remeasured under the new speed-aware runner.
